# システム構成の概要

この設定は、Raspberry PiからPCにビデオをストリーミングし、PCからRaspberry Piに制御コマンドを送信することを可能にする、有線ネットワーク接続を前提としています。

## 1. Raspberry Pi サーバー (`only_rasp1/pi_server.py`)

*   **役割:** サーバーとして機能し、カメラからビデオをキャプチャし、制御コマンドを受信します。
*   **ビデオストリーミング (UDP - ポート 5005):**
    *   `cv2.VideoCapture(0)` を使用してRaspberry Piのカメラにアクセスします。
    *   ビデオフレームをキャプチャし、JPEG形式 (品質50%) に圧縮し、設定されたPC_IP (例: "192.168.50.10") の `VIDEO_PORT` (5005) にUDPパケットとして送信します。
    *   軽量化のため、フレームは320x240にリサイズされます。
    *   約30 FPSにフレームレートを制限するために `time.sleep(0.03)` が含まれています。
    *   ビデオ送信は別のスレッドで実行されます。
*   **制御コマンド受信 (TCP - ポート 5006):**
    *   `MY_IP` ("0.0.0.0", すべてのインターフェース) および `CONTROL_PORT` (5006) で着信TCP接続をリッスンします。
    *   接続後、連続してデータを受信します。
    *   受信したデータをUTF-8でデコードし、改行で分割して1つのパケット内の複数のコマンドを処理します。
    *   "FORWARD"、"BACK"、"LEFT"、"RIGHT"、"STOP" などのコマンドを処理します。
    *   **主な機能:** 受信したコマンドが `last_command` と異なる場合にのみコマンドを処理および表示し、状態ベースの制御 (例: 現在プレースホルダーであるGPIO操作) を最適化します。
    *   制御受信はメインスレッドで実行されます。

## 2. PC クライアント (`RaspberryPi/pc_client.py`)

*   **役割:** クライアントとして機能し、Raspberry Piから受信したビデオを表示し、制御コマンドを送信します。
*   **ビデオ受信と表示 (UDP - ポート 5005):**
    *   `MY_IP` ("0.0.0.0") および `VIDEO_PORT` (5005) でUDPパケットをリッスンします。
    *   圧縮されたJPEGデータを受信し、`cv2.imdecode` を使用してビデオフレームにデコードし、「Raspberry Pi Camera (Low Latency)」というウィンドウに表示します。
    *   ビデオ受信は別のスレッドで実行されます。
    *   'q' を押すとビデオウィンドウが閉じます。
*   **制御コマンド送信 (TCP - ポート 5006):**
    *   `RPI_IP` (例: "192.168.50.20") の `CONTROL_PORT` (5006) にTCP接続を確立しようとします。
    *   `pygame` を使用してキーボード入力 (WASDキー) を検出します。
    *   キー入力に基づいて:
        *   'W' は "FORWARD" を送信します。
        *   'S' は "BACK" を送信します。
        *   'A' は "LEFT" を送信します。
        *   'D' は "RIGHT" を送信します。
        *   移動キーが押されていない場合、"STOP" が送信されます。
    *   コマンド (改行文字が付加されたもの) をUTF-8でエンコードして送信します。
    *   コマンドの頻繁な送信を防ぐために `time.sleep(0.05)` が含まれています。
    *   制御送信はメインスレッドで実行されます。

## 3. Raspberry Pi Server 2 (`control_main/rasp2/ras2_pi_server.py`)

*   **役割:** サーバーとして機能し、カメラからのビデオストリーミングと、より高度なモーターおよびサーボ制御コマンドの受信を行います。
*   **ビデオストリーミング (UDP - ポート 5005):**
    *   `cv2.VideoCapture(0)` を使用してRaspberry Piのカメラにアクセスします。
    *   カメラの解像度を 640x480 に設定し、フレームをキャプチャします。
    *   `SHARED_STATE["tilt"]` の値に応じて、ビデオフレームを 180 度回転させます（例：カメラが上向きに傾いた際に、画面表示を正しく保つため）。
    *   ビデオフレームは JPEG 形式 (品質 50%) に圧縮され、設定された `PC_IP` (例: "192.168.50.10") の `VIDEO_PORT` (5005) に UDP パケットとして送信されます。
    *   UDP パケットサイズが 65000 バイトを超える場合は送信をスキップし、データオーバーフローを防ぎます。
    *   約 30 FPS にフレームレートを制限するために `time.sleep(0.03)` が含まれています。
    *   ビデオ送信は別のスレッドで実行されます。
*   **制御コマンド受信 (TCP - ポート 5006):**
    *   `MY_IP` ("0.0.0.0", すべてのインターフェース) および `CONTROL_PORT` (5006) で着信 TCP 接続をリッスンします。
    *   `pigpio` ライブラリを使用して、GPIO ピンを介してモーターとサーボを制御します。
    *   **モーター制御:**
        *   左モーター: PWMピン GPIO 12, DIRピン GPIO 20
        *   右モーター: PWMピン GPIO 13, DIRピン GPIO 21
        *   速度スケーリング (`SPEED_SCALE = 0.7`)、PWM 周波数 (20000Hz)、デッドゾーン (0.15) が設定されています。
    *   **サーボ制御:**
        *   チルト（上下）サーボ: GPIO 18
        *   パン（左右）サーボ: GPIO 19
        *   サーボの速度 (`SERVO_SPEED_TILT`, `SERVO_SPEED_PAN`) と、角度に基づいた PWM パルス幅 (500-2500μs for 0-180°) が設定されています。
    *   **制御コマンド:** JSON 形式で、`controller` オブジェクト内に `LS_Y` (左スティック Y)、`RS_Y` (右スティック Y)、`HAT_X` (十字キー X)、`HAT_Y` (十字キー Y) の値が含まれます。
        *   `LS_Y` は右モーターの速度を制御します。
        *   `RS_Y` は左モーターの速度を制御します。
        *   `HAT_Y` はチルトサーボ（カメラ上下）を制御します。
        *   `HAT_X` はパンサーボ（カメラ左右）を制御します。
    *   `SHARED_STATE["tilt"]` を更新し、ビデオストリームに反映させます。
    *   制御受信はメインスレッドで実行されます。

## 4. PC クライアント 2 (`control_main/rasp2/ras2_pc_client.py`)

*   **役割:** クライアントとして機能し、Raspberry Pi 2 から受信したビデオを表示し、キーボード入力を介して制御コマンドを送信します。
*   **ビデオ受信と表示 (UDP - ポート 5005):**
    *   `MY_IP` ("0.0.0.0") および `VIDEO_PORT` (5005) で UDP パケットをリッスンします。
    *   圧縮された JPEG データを受信し、`cv2.imdecode` を使用してビデオフレームにデコードします。
    *   ビデオフレームは `VIDEO_ROTATION = 270` 設定に従って回転され、「Camera View (Press Q in Control Window to Quit)」というウィンドウに表示されます。
    *   ビデオ受信は別のスレッドで実行されます。
*   **制御コマンド送信 (TCP - ポート 5006):**
    *   `RPI_IP` (例: "192.168.50.20") の `CONTROL_PORT` (5006) に TCP 接続を試みます。
    *   `pygame` を使用してキーボード入力を検出します。
    *   **キー入力マッピング:**
        *   `W`: 左モーター前進 (`LS_Y: -1.0`)
        *   `S`: 左モーター後退 (`LS_Y: 1.0`)
        *   `O`: 右モーター前進 (`RS_Y: -1.0`)
        *   `L`: 右モーター後退 (`RS_Y: 1.0`)
        *   `3`: カメラ上 (`HAT_Y: 1`)
        *   `4`: カメラ下 (`HAT_Y: -1`)
        *   `8`: カメラ左 (`HAT_X: -1`)
        *   `9`: カメラ右 (`HAT_X: 1`)
        *   `Q`: プログラム終了
    *   コマンドは JSON 形式で、`controller` オブジェクト内に `LS_Y`, `RS_Y`, `HAT_X`, `HAT_Y` の値が含まれて送信されます。
    *   コマンドの頻繁な送信を防ぐために、前回の送信コマンドと異なる場合にのみ送信し、ログにも表示します。
    *   制御送信はメインスレッドで実行されます。
    *   操作方法が表示される Pygame ウィンドウは、クリックしてフォーカスする必要があります。

## 変更ログ

### 2025年12月8日月曜日

#### 目的:
`windows/control/control_GUI.py` で実装されているコントローラー入力要素を `RaspberryPi/pc_client.py` に反映させ、`only_rasp1/pi_server.py` でそれらのコマンドを処理できるようにする。

#### 変更点:

1.  **`RaspberryPi/pc_client.py` の更新:**
    *   `windows/control/control_GUI.py` から `AXIS_MAPPING`, `BUTTON_MAPPING`, `HAT_MAPPING` のマッピング情報をコピーして追加しました。
    *   `send_control` 関数を以下のように変更しました:
        *   `pygame.joystick.init()` を追加し、ジョイスティックの検出と初期化を行いました。
        *   ジョイスティックが接続されていない場合や、接続・切断された場合のメッセージ表示および再初期化ロジックを追加しました (`pygame.JOYDEVICEADDED`, `pygame.JOYDEVICEREMOVED` イベント処理)。
        *   ジョイスティックの左スティック (`axis 0`, `axis 1`)、十字キー (`hat 0`)、ボタン (`button 0`～`3`など)、およびトリガー (`axis 4`, `axis 5`) の入力をポーリングし、それらに基づいて以下の新しいコマンドを生成するようにしました:
            *   アナログスティック: `FORWARD`, `BACK`, `LEFT`, `RIGHT` (閾値 `-0.5` / `0.5` で判断)
            *   十字キー: `DPAD_UP`, `DPAD_DOWN`, `DPAD_LEFT`, `DPAD_RIGHT`
            *   ボタン: `BUTTON_A`, `BUTTON_B`, `BUTTON_X`, `BUTTON_Y`
            *   トリガー: `TRIGGER_LT`, `TRIGGER_RT` (閾値 `0.5` で判断)
        *   `last_command` 変数を導入し、前回の送信コマンドと異なる場合にのみTCPでコマンドを送信するように最適化しました。これにより、Raspberry Pi側での不要な重複処理を軽減します。
        *   既存のWASDキー入力は、ジョイスティックからの明示的なコマンドがない（`current_command == "STOP"` の状態）場合にのみ有効となるように調整し、ジョイスティックとキーボードの併用、またはコントローラーが接続されていない場合の代替操作を可能にしました。

2.  **`only_rasp1/pi_server.py` の更新:**
    *   `receive_control` 関数内のGPIO制御ブロックに、`pc_client.py` から送信される可能性のある新しいコマンド (`DPAD_UP`, `DPAD_DOWN`, `DPAD_LEFT`, `DPAD_RIGHT`, `BUTTON_A`, `BUTTON_B`, `BUTTON_X`, `BUTTON_Y`, `TRIGGER_LT`, `TRIGGER_RT`) に対する `elif` 文のプレースホルダーを追加しました。
    *   これらのコマンドに対する具体的なGPIO操作は、現在のところ `pass` としています。

#### 影響:
*   PCクライアントは、ゲームコントローラー（検出された場合）またはキーボード（WASD）からの入力に基づいて、より詳細な制御コマンドをRaspberry Piに送信できるようになりました。
*   Raspberry Piサーバーは、これらの新しいコマンドを受信し、それぞれのコマンドに対応するGPIOアクションをトリガーする準備ができました（実際のGPIO実装はまだです）。
*   コマンド送信の効率が向上し、Raspberry Pi側での重複した状態変更の処理が削減されます。
